01 / ARCHITECTURE

Tenant isolation that holds under load

A shared PostgreSQL database, with every record carrying a tenant_id and Row-Level Security enforcing it at the database layer — not just in application code. That's what keeps one agency's data from ever touching another's, even under bugs or bad joins.

🔒 Row-Level Security scoped by tenant_id on every table
01

Fast, safe cross-tenant reporting

Materialized views for the Master Admin dashboard aggregate across every agency without opening a gap in tenant-level security or slowing down day-to-day queries.

02

Referential integrity by design

Every table — clients, meters, utility_bills, interval_data — carries its own foreign key to the parent record and its own tenant_id for fast, indexed filtering.

03

Role-based access as it grows

Proper indexing and role-based permissions from day one, so adding the next agency or the next meter doesn't mean revisiting the schema.

04 / FRONTEND CLEANUP

Removing features without breaking the backend

If part of the job is trimming chart-only components from the React frontend, I don't start by deleting code.

Step 1

Map before touching

Identify which components are purely chart rendering, and map the API endpoints each one actually consumes.

Step 2

Flag, don't delete

Disable components behind feature flags or route-level conditions first, keeping API calls intact wherever they're used elsewhere.

Step 3

Regression test, then clean

Verify auth, navigation, dashboards, and integrations still work — only then remove the unused endpoints and dead code.
